# <a name="index-table-pattern"></a><span data-ttu-id="e9234-104">索引表模式</span><span class="sxs-lookup"><span data-stu-id="e9234-104">Index Table pattern</span></span>

[!INCLUDE [header](../_includes/header.md)]

<span data-ttu-id="e9234-105">基于数据存储中经常由查询引用的字段创建索引。</span><span class="sxs-lookup"><span data-stu-id="e9234-105">Create indexes over the fields in data stores that are frequently referenced by queries.</span></span> <span data-ttu-id="e9234-106">此模式可让应用程序更快地找到要从数据存储中检索的数据，从而提高了查询性能。</span><span class="sxs-lookup"><span data-stu-id="e9234-106">This pattern can improve query performance by allowing applications to more quickly locate the data to retrieve from a data store.</span></span>

## <a name="context-and-problem"></a><span data-ttu-id="e9234-107">上下文和问题</span><span class="sxs-lookup"><span data-stu-id="e9234-107">Context and problem</span></span>

<span data-ttu-id="e9234-108">许多数据存储使用主键组织实体集合的数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-108">Many data stores organize the data for a collection of entities using the primary key.</span></span> <span data-ttu-id="e9234-109">应用程序可以使用此键来查找和检索数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-109">An application can use this key to locate and retrieve data.</span></span> <span data-ttu-id="e9234-110">下图显示了一个保存客户信息的数据存储示例。</span><span class="sxs-lookup"><span data-stu-id="e9234-110">The figure shows an example of a data store holding customer information.</span></span> <span data-ttu-id="e9234-111">主键为“客户 ID”。</span><span class="sxs-lookup"><span data-stu-id="e9234-111">The primary key is the Customer ID.</span></span> <span data-ttu-id="e9234-112">该图显示按主键（“客户 ID”）组织的客户信息。</span><span class="sxs-lookup"><span data-stu-id="e9234-112">The figure shows customer information organized by the primary key (Customer ID).</span></span>

![图 1 - 按主键（“客户 ID”）组织的客户信息。](./_images/index-table-figure-1.png)

<span data-ttu-id="e9234-114">尽管主键对于基于此键值提取数据的查询很有作用，但如果应用程序需要基于其他某个字段检索数据，则可能无法使用主键。</span><span class="sxs-lookup"><span data-stu-id="e9234-114">While the primary key is valuable for queries that fetch data based on the value of this key, an application might not be able to use the primary key if it needs to retrieve data based on some other field.</span></span> <span data-ttu-id="e9234-115">在客户示例中，如果应用程序只是通过引用其他某个属性（例如，客户所在的城镇）的值查询数据，则无法使用“客户 ID”主键来检索客户。</span><span class="sxs-lookup"><span data-stu-id="e9234-115">In the customers example, an application can't use the Customer ID primary key to retrieve customers if it queries data solely by referencing the value of some other attribute, such as the town in which the customer is located.</span></span> <span data-ttu-id="e9234-116">若要执行此类查询，应用程序可能需要提取并检查每条客户记录，而这是一个缓慢的过程。</span><span class="sxs-lookup"><span data-stu-id="e9234-116">To perform a query such as this, the application might have to fetch and examine every customer record, which could be a slow process.</span></span>

<span data-ttu-id="e9234-117">许多关系数据库管理系统支持辅助索引。</span><span class="sxs-lookup"><span data-stu-id="e9234-117">Many relational database management systems support secondary indexes.</span></span> <span data-ttu-id="e9234-118">辅助索引是按一个或多个非主键（辅助键）字段组织的独立数据结构，指示每个索引值的数据的存储位置。</span><span class="sxs-lookup"><span data-stu-id="e9234-118">A secondary index is a separate data structure that's organized by one or more nonprimary (secondary) key fields, and it indicates where the data for each indexed value is stored.</span></span> <span data-ttu-id="e9234-119">辅助索引中的项通常已按辅助键的值排序，以便快速查找数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-119">The items in a secondary index are typically sorted by the value of the secondary keys to enable fast lookup of data.</span></span> <span data-ttu-id="e9234-120">这些索引通常由数据库管理系统自动维护。</span><span class="sxs-lookup"><span data-stu-id="e9234-120">These indexes are usually maintained automatically by the database management system.</span></span>

<span data-ttu-id="e9234-121">可以根据需要创建任意数目的辅助索引来支持应用程序执行的不同查询。</span><span class="sxs-lookup"><span data-stu-id="e9234-121">You can create as many secondary indexes as you need to support the different queries that your application performs.</span></span> <span data-ttu-id="e9234-122">例如，在关系数据库中“客户 ID”是主键的“客户”表中，如果应用程序经常根据客户所在的城镇查找客户，则基于城镇字段添加一个辅助索引会很有作用。</span><span class="sxs-lookup"><span data-stu-id="e9234-122">For example, in a Customers table in a relational database where the Customer ID is the primary key, it's beneficial to add a secondary index over the town field if the application frequently looks up customers by the town where they reside.</span></span>

<span data-ttu-id="e9234-123">尽管辅助索引在关系数据库系统中很常见，但云应用程序使用的大多数 NoSQL 数据存储并不提供等效的功能。</span><span class="sxs-lookup"><span data-stu-id="e9234-123">However, although secondary indexes are common in relational systems, most NoSQL data stores used by cloud applications don't provide an equivalent feature.</span></span>

## <a name="solution"></a><span data-ttu-id="e9234-124">解决方案</span><span class="sxs-lookup"><span data-stu-id="e9234-124">Solution</span></span>

<span data-ttu-id="e9234-125">如果数据存储不支持辅助索引，你可以通过创建自己的索引表来手动模拟辅助索引。</span><span class="sxs-lookup"><span data-stu-id="e9234-125">If the data store doesn't support secondary indexes, you can emulate them manually by creating your own index tables.</span></span> <span data-ttu-id="e9234-126">索引表按指定的键组织数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-126">An index table organizes the data by a specified key.</span></span> <span data-ttu-id="e9234-127">根据所需的辅助索引数目以及应用程序执行的查询性质，我们通常可以使用三种策略来构建索引表。</span><span class="sxs-lookup"><span data-stu-id="e9234-127">Three strategies are commonly used for structuring an index table, depending on the number of secondary indexes that are required and the nature of the queries that an application performs.</span></span>

<span data-ttu-id="e9234-128">第一种策略是复制每个索引表中的数据，但按不同的键（完全非规范化）组织这些数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-128">The first strategy is to duplicate the data in each index table but organize it by different keys (complete denormalization).</span></span> <span data-ttu-id="e9234-129">下图显示了按“城镇”和“姓氏”组织相同客户信息的索引表。</span><span class="sxs-lookup"><span data-stu-id="e9234-129">The next figure shows index tables that organize the same customer information by Town and LastName.</span></span>

![图 2 - 已复制每个索引表中的数据](./_images/index-table-figure-2.png)

<span data-ttu-id="e9234-131">如果相对于使用每个键查询数据的次数而言数据较为静态，则此策略适用。</span><span class="sxs-lookup"><span data-stu-id="e9234-131">This strategy is appropriate if the data is relatively static compared to the number of times it's queried using each key.</span></span> <span data-ttu-id="e9234-132">如果数据较为动态，则维护每个索引表所产生的处理开销会变得过大，因此，此方法并不有效。</span><span class="sxs-lookup"><span data-stu-id="e9234-132">If the data is more dynamic, the processing overhead of maintaining each index table becomes too large for this approach to be useful.</span></span> <span data-ttu-id="e9234-133">此外，如果数据量极大，存储重复数据所需的空间量也很大。</span><span class="sxs-lookup"><span data-stu-id="e9234-133">Also, if the volume of data is very large, the amount of space required to store the duplicate data is significant.</span></span>

<span data-ttu-id="e9234-134">第二种策略是创建按不同键组织的规范化索引表，并使用主键来引用原始数据，而不是复制数据，如下图所示。</span><span class="sxs-lookup"><span data-stu-id="e9234-134">The second strategy is to create normalized index tables organized by different keys and reference the original data by using the primary key rather than duplicating it, as shown in the following figure.</span></span> <span data-ttu-id="e9234-135">原始数据称为事实数据表。</span><span class="sxs-lookup"><span data-stu-id="e9234-135">The original data is called a fact table.</span></span>

![图 3 - 数据由每个索引表引用](./_images/index-table-figure-3.png)

<span data-ttu-id="e9234-137">此方法可节省空间，并减少维护重复数据所产生的开销。</span><span class="sxs-lookup"><span data-stu-id="e9234-137">This technique saves space and reduces the overhead of maintaining duplicate data.</span></span> <span data-ttu-id="e9234-138">其缺点在于，应用程序必须使用辅助键执行两项查找操作才能找到数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-138">The disadvantage is that an application has to perform two lookup operations to find data using a secondary key.</span></span> <span data-ttu-id="e9234-139">应用程序必须在索引表中查找数据的主键，然后使用主键来查找事实数据表中的数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-139">It has to find the primary key for the data in the index table, and then use the primary key to look up the data in the fact table.</span></span>

<span data-ttu-id="e9234-140">第三种策略是创建按不同键组织的、可复制经常检索的字段的部分规范化索引表。</span><span class="sxs-lookup"><span data-stu-id="e9234-140">The third strategy is to create partially normalized index tables organized by different keys that duplicate frequently retrieved fields.</span></span> <span data-ttu-id="e9234-141">引用事实数据表即可访问不经常访问的字段。</span><span class="sxs-lookup"><span data-stu-id="e9234-141">Reference the fact table to access less frequently accessed fields.</span></span> <span data-ttu-id="e9234-142">下图显示了如何在每个索引表中复制经常访问的数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-142">The next figure shows how commonly accessed data is duplicated in each index table.</span></span>

![图 4 - 在每个索引表中复制经常访问的数据。](./_images/index-table-figure-4.png)

<span data-ttu-id="e9234-144">如果使用此策略，可以在前两种方法之间取得平衡。</span><span class="sxs-lookup"><span data-stu-id="e9234-144">With this strategy, you can strike a balance between the first two approaches.</span></span> <span data-ttu-id="e9234-145">可以使用单个查找操作快速检索用于常见查询的数据，同时，空间和维护开销不如复制整个数据集那么大。</span><span class="sxs-lookup"><span data-stu-id="e9234-145">The data for common queries can be retrieved quickly by using a single lookup, while the space and maintenance overhead isn't as significant as duplicating the entire data set.</span></span>

<span data-ttu-id="e9234-146">如果应用程序经常通过指定值的组合（例如，“查找居住在 Redmond，且姓氏为 Smith 的所有客户”）查询数据，则你可以针对索引表中的项实现“城镇”属性与“姓氏”属性串联形式的键。</span><span class="sxs-lookup"><span data-stu-id="e9234-146">If an application frequently queries data by specifying a combination of values (for example, “Find all customers that live in Redmond and that have a last name of Smith”), you could implement the keys to the items in the index table as a concatenation of the Town attribute and the LastName attribute.</span></span> <span data-ttu-id="e9234-147">下图显示了基于组合键的索引表。</span><span class="sxs-lookup"><span data-stu-id="e9234-147">The next figure shows an index table based on composite keys.</span></span> <span data-ttu-id="e9234-148">这些键先按“城镇”排序，然后，针对具有相同“城镇”值的记录按“姓氏”排序。</span><span class="sxs-lookup"><span data-stu-id="e9234-148">The keys are sorted by Town, and then by LastName for records that have the same value for Town.</span></span>

![图 5 - 基于组合键的索引表](./_images/index-table-figure-5.png)

<span data-ttu-id="e9234-150">索引表可以加速分片数据的查询操作，当分片键已经过哈希处理时特别有用。</span><span class="sxs-lookup"><span data-stu-id="e9234-150">Index tables can speed up query operations over sharded data, and are especially useful where the shard key is hashed.</span></span> <span data-ttu-id="e9234-151">下图显示了将“客户 ID”哈希用作分片键的示例。</span><span class="sxs-lookup"><span data-stu-id="e9234-151">The next figure shows an example where the shard key is a hash of the Customer ID.</span></span> <span data-ttu-id="e9234-152">索引表可按未经哈希处理的值（“城镇”和“姓氏”）组织数据，并提供经过哈希处理的分片键作为查找数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-152">The index table can organize data by the nonhashed value (Town and LastName), and provide the hashed shard key as the lookup data.</span></span> <span data-ttu-id="e9234-153">这样，在应用程序需要检索处于某个范围内的数据，或者需要根据未经哈希处理的键的顺序提取数据时，就无需反复计算哈希键（此操作的开销较高）。</span><span class="sxs-lookup"><span data-stu-id="e9234-153">This can save the application from repeatedly calculating hash keys (an expensive operation) if it needs to retrieve data that falls within a range, or it needs to fetch data in order of the nonhashed key.</span></span> <span data-ttu-id="e9234-154">例如，通过在索引表中找到匹配项（所有项存储在相邻的块中），可以快速解决类似于“查找居住在 Redmond 的所有客户”的查询。</span><span class="sxs-lookup"><span data-stu-id="e9234-154">For example, a query such as “Find all customers that live in Redmond” can be quickly resolved by locating the matching items in the index table, where they're all stored in a contiguous block.</span></span> <span data-ttu-id="e9234-155">然后，可以使用索引表中存储的分片键，遵循对客户数据的引用进行操作。</span><span class="sxs-lookup"><span data-stu-id="e9234-155">Then, follow the references to the customer data using the shard keys stored in the index table.</span></span>

![图 6 - 提供快速分片数据查找的索引表](./_images/index-table-figure-6.png)

## <a name="issues-and-considerations"></a><span data-ttu-id="e9234-157">问题和注意事项</span><span class="sxs-lookup"><span data-stu-id="e9234-157">Issues and considerations</span></span>

<span data-ttu-id="e9234-158">在决定如何实现此模式时，请考虑以下几点：</span><span class="sxs-lookup"><span data-stu-id="e9234-158">Consider the following points when deciding how to implement this pattern:</span></span>

- <span data-ttu-id="e9234-159">维护辅助索引所产生的开销可能很大。</span><span class="sxs-lookup"><span data-stu-id="e9234-159">The overhead of maintaining secondary indexes can be significant.</span></span> <span data-ttu-id="e9234-160">必须分析并了解应用程序使用的查询。</span><span class="sxs-lookup"><span data-stu-id="e9234-160">You must analyze and understand the queries that your application uses.</span></span> <span data-ttu-id="e9234-161">仅当应用程序定期使用索引表时，才创建索引表。</span><span class="sxs-lookup"><span data-stu-id="e9234-161">Only create index tables when they're likely to be used regularly.</span></span> <span data-ttu-id="e9234-162">不要创建推理索引表来支持应用程序不会执行或只是偶尔执行的查询。</span><span class="sxs-lookup"><span data-stu-id="e9234-162">Don't create speculative index tables to support queries that an application doesn't perform, or performs only occasionally.</span></span>
- <span data-ttu-id="e9234-163">复制索引表中的数据可能会明显增大存储成本方面的开销，以及维护多个数据副本所需的工作量。</span><span class="sxs-lookup"><span data-stu-id="e9234-163">Duplicating data in an index table can add significant overhead in storage costs and the effort required to maintain multiple copies of data.</span></span>
- <span data-ttu-id="e9234-164">将索引表实现为引用原始数据的规范化结构需要应用程序执行两项查找操作才能找到数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-164">Implementing an index table as a normalized structure that references the original data requires an application to perform two lookup operations to find data.</span></span> <span data-ttu-id="e9234-165">第一项操作搜索索引表以检索主键，第二项操作使用主键提取数据。</span><span class="sxs-lookup"><span data-stu-id="e9234-165">The first operation searches the index table to retrieve the primary key, and the second uses the primary key to fetch the data.</span></span>
- <span data-ttu-id="e9234-166">如果系统基于极大型数据集整合了许多索引表，则可能很难维持索引表与原始数据之间的一致性。</span><span class="sxs-lookup"><span data-stu-id="e9234-166">If a system incorporates a number of index tables over very large data sets, it can be difficult to maintain consistency between index tables and the original data.</span></span> <span data-ttu-id="e9234-167">也许可以围绕最终一致性模型设计应用程序。</span><span class="sxs-lookup"><span data-stu-id="e9234-167">It might be possible to design the application around the eventual consistency model.</span></span> <span data-ttu-id="e9234-168">例如，若要插入、更新或删除数据，应用程序可以向队列发布一条消息，让单独的任务执行该操作，并维护以异步方式引用此数据的索引表。</span><span class="sxs-lookup"><span data-stu-id="e9234-168">For example, to insert, update, or delete data, an application could post a message to a queue and let a separate task perform the operation and maintain the index tables that reference this data asynchronously.</span></span> <span data-ttu-id="e9234-169">有关实施最终一致性的详细信息，请参阅[数据一致性入门](https://msdn.microsoft.com/library/dn589800.aspx)。</span><span class="sxs-lookup"><span data-stu-id="e9234-169">For more information about implementing eventual consistency, see the [Data Consistency Primer](https://msdn.microsoft.com/library/dn589800.aspx).</span></span>

   >  <span data-ttu-id="e9234-170">Microsoft Azure 存储表支持对同一个分区中保存的数据所发生的更改执行事务更新（称为实体组事务）。</span><span class="sxs-lookup"><span data-stu-id="e9234-170">Microsoft Azure storage tables support transactional updates for changes made to data held in the same partition (referred to as entity group transactions).</span></span> <span data-ttu-id="e9234-171">如果可以在同一个分区中存储一个事实数据表以及一个或多个索引表的数据，则可以使用此功能来帮助确保一致性。</span><span class="sxs-lookup"><span data-stu-id="e9234-171">If you can store the data for a fact table and one or more index tables in the same partition, you can use this feature to help ensure consistency.</span></span>

- <span data-ttu-id="e9234-172">索引表本身可能已分区或分片。</span><span class="sxs-lookup"><span data-stu-id="e9234-172">Index tables might themselves be partitioned or sharded.</span></span>

## <a name="when-to-use-this-pattern"></a><span data-ttu-id="e9234-173">何时使用此模式</span><span class="sxs-lookup"><span data-stu-id="e9234-173">When to use this pattern</span></span>

<span data-ttu-id="e9234-174">如果应用程序经常需要使用除主键（或分片键）以外的某个键检索数据，则使用此模式可提高查询性能。</span><span class="sxs-lookup"><span data-stu-id="e9234-174">Use this pattern to improve query performance when an application frequently needs to retrieve data by using a key other than the primary (or shard) key.</span></span>

<span data-ttu-id="e9234-175">在以下情况下，此模式可能不起作用：</span><span class="sxs-lookup"><span data-stu-id="e9234-175">This pattern might not be useful when:</span></span>

- <span data-ttu-id="e9234-176">数据是易失性的。</span><span class="sxs-lookup"><span data-stu-id="e9234-176">Data is volatile.</span></span> <span data-ttu-id="e9234-177">索引表可能很快过时，因此变得低效，或者维护索引表所产生的开销超过了它所带来的成本节省。</span><span class="sxs-lookup"><span data-stu-id="e9234-177">An index table can become out of date very quickly, making it ineffective or making the overhead of maintaining the index table greater than any savings made by using it.</span></span>
- <span data-ttu-id="e9234-178">选作索引表辅助键的字段没有辨别性，可能只包含少量的值（例如性别）。</span><span class="sxs-lookup"><span data-stu-id="e9234-178">A field selected as the secondary key for an index table is nondiscriminating and can only have a small set of values (for example, gender).</span></span>
