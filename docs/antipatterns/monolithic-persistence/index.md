# <a name="monolithic-persistence-antipattern"></a><span data-ttu-id="c2897-103">整体持久性反模式</span><span class="sxs-lookup"><span data-stu-id="c2897-103">Monolithic Persistence antipattern</span></span>

<span data-ttu-id="c2897-104">将应用程序的所有数据放入单个数据存储可能会降低性能，原因是这会导致资源争用，或者数据存储不很适合某些数据。</span><span class="sxs-lookup"><span data-stu-id="c2897-104">Putting all of an application's data into a single data store can hurt performance, either because it leads to resource contention, or because the data store is not a good fit for some of the data.</span></span>

## <a name="problem-description"></a><span data-ttu-id="c2897-105">问题描述</span><span class="sxs-lookup"><span data-stu-id="c2897-105">Problem description</span></span>

<span data-ttu-id="c2897-106">一直以来，不管应用程序需要存储哪些不同类型的数据，往往都只使用单个数据存储。</span><span class="sxs-lookup"><span data-stu-id="c2897-106">Historically, applications have often used a single data store, regardless of the different types of data that the application might need to store.</span></span> <span data-ttu-id="c2897-107">这样做的原因通常是为了简化应用程序设计，或者受限于开发团队的现有技能组合。</span><span class="sxs-lookup"><span data-stu-id="c2897-107">Usually this was done to simplify the application design, or else to match the existing skill set of the development team.</span></span>

<span data-ttu-id="c2897-108">现代基于云的系统往往附带其他功能性和非功能性要求，需要存储许多异构类型的数据，例如文档、图像、缓存数据、排队消息、应用程序日志和遥测数据。</span><span class="sxs-lookup"><span data-stu-id="c2897-108">Modern cloud-based systems often have additional functional and nonfunctional requirements, and need to store many heterogenous types of data, such as documents, images, cached data, queued messages, application logs, and telemetry.</span></span> <span data-ttu-id="c2897-109">遵循传统方法将所有这些信息放入同一个数据存储可能会损害性能，主要原因有两个：</span><span class="sxs-lookup"><span data-stu-id="c2897-109">Following the traditional approach and putting all of this information into the same data store can hurt performance, for two main reasons:</span></span>

- <span data-ttu-id="c2897-110">在同一个数据存储中存储和检索大量不相关的数据可能导致资源争用，从而导致响应时间增加和连接失败。</span><span class="sxs-lookup"><span data-stu-id="c2897-110">Storing and retrieving large amounts of unrelated data in the same data store can cause contention, which in turn leads to slow response times and connection failures.</span></span>
- <span data-ttu-id="c2897-111">不管选择哪个数据存储，它都不一定最适合所有不同类型的数据，或者未针对应用程序执行的操作进行优化。</span><span class="sxs-lookup"><span data-stu-id="c2897-111">Whichever data store is chosen, it might not be the best fit for all of the different types of data, or it might not be optimized for the operations that the application performs.</span></span>

<span data-ttu-id="c2897-112">以下示例演示了一个向数据库添加新记录，并将结果记录到日志的 ASP.NET Web API 控制器。</span><span class="sxs-lookup"><span data-stu-id="c2897-112">The following example shows an ASP.NET Web API controller that adds a new record to a database and also records the result to a log.</span></span> <span data-ttu-id="c2897-113">日志保存在业务数据所在的同一个数据库中。</span><span class="sxs-lookup"><span data-stu-id="c2897-113">The log is held in the same database as the business data.</span></span> <span data-ttu-id="c2897-114">可在[此处][sample-app]找到完整示例。</span><span class="sxs-lookup"><span data-stu-id="c2897-114">You can find the complete sample [here][sample-app].</span></span>

```csharp
public class MonoController : ApiController
{
    private static readonly string ProductionDb = ...;

    public async Task<IHttpActionResult> PostAsync([FromBody]string value)
    {
        await DataAccess.InsertPurchaseOrderHeaderAsync(ProductionDb);
        await DataAccess.LogAsync(ProductionDb, LogTableName);
        return Ok();
    }
}
```

<span data-ttu-id="c2897-115">日志记录的生成速率可能会影响业务操作的性能。</span><span class="sxs-lookup"><span data-stu-id="c2897-115">The rate at which log records are generated will probably affect the performance of the business operations.</span></span> <span data-ttu-id="c2897-116">如果另一个组件（例如应用程序进程监视器）定期读取和处理日志数据，则它们也可能影响业务操作。</span><span class="sxs-lookup"><span data-stu-id="c2897-116">And if another component, such as an application process monitor, regularly reads and processes the log data, that can also affect the business operations.</span></span>

## <a name="how-to-fix-the-problem"></a><span data-ttu-id="c2897-117">如何解决问题</span><span class="sxs-lookup"><span data-stu-id="c2897-117">How to fix the problem</span></span>

<span data-ttu-id="c2897-118">根据数据的用法隔离数据。</span><span class="sxs-lookup"><span data-stu-id="c2897-118">Separate data according to its use.</span></span> <span data-ttu-id="c2897-119">对于每个数据集，选择最符合数据集用法的数据存储。</span><span class="sxs-lookup"><span data-stu-id="c2897-119">For each data set, select a data store that best matches how that data set will be used.</span></span> <span data-ttu-id="c2897-120">在前面的示例中，应用程序应将日志记录到与保存业务数据的数据库不同的存储：</span><span class="sxs-lookup"><span data-stu-id="c2897-120">In the previous example, the application should be logging to a separate store from the database that holds business data:</span></span>

```csharp

{
    private static readonly string ProductionDb = ...;
    private static readonly string LogDb = ...;

    public async Task<IHttpActionResult> PostAsync([FromBody]string value)
    {
        await DataAccess.InsertPurchaseOrderHeaderAsync(ProductionDb);
        // Log to a different data store.
        await DataAccess.LogAsync(LogDb, LogTableName);
        return Ok();
    }
}
```

## <a name="considerations"></a><span data-ttu-id="c2897-121">注意事项</span><span class="sxs-lookup"><span data-stu-id="c2897-121">Considerations</span></span>

- <span data-ttu-id="c2897-122">根据数据的使用方式及其访问方式隔离数据。</span><span class="sxs-lookup"><span data-stu-id="c2897-122">Separate data by the way it is used and how it is accessed.</span></span> <span data-ttu-id="c2897-123">例如，不要将日志信息和业务数据存储在同一个数据存储中。</span><span class="sxs-lookup"><span data-stu-id="c2897-123">For example, don't store log information and business data in the same data store.</span></span> <span data-ttu-id="c2897-124">这些数据类型的要求和访问模式明显不同。</span><span class="sxs-lookup"><span data-stu-id="c2897-124">These types of data have significantly different requirements and patterns of access.</span></span> <span data-ttu-id="c2897-125">日志记录原生是连续的，而业务数据更有可能需要随机访问，因此通常是关系型的。</span><span class="sxs-lookup"><span data-stu-id="c2897-125">Log records are inherently sequential, while business data is more likely to require random access, and is often relational.</span></span>

- <span data-ttu-id="c2897-126">考虑每种数据类型的数据访问模式。</span><span class="sxs-lookup"><span data-stu-id="c2897-126">Consider the data access pattern for each type of data.</span></span> <span data-ttu-id="c2897-127">例如，将带有格式的报告和文档存储在 [Cosmos DB][CosmosDB] 等文档数据库中，但使用 [Azure Redis 缓存][Azure-cache]来缓存临时数据。</span><span class="sxs-lookup"><span data-stu-id="c2897-127">For example, store formatted reports and documents in a document database such as [Cosmos DB][CosmosDB], but use [Azure Redis Cache][Azure-cache] to cache temporary data.</span></span>

- <span data-ttu-id="c2897-128">如果遵循了此指导原则，但仍达到了数据库限制，可能需要纵向扩展数据库。</span><span class="sxs-lookup"><span data-stu-id="c2897-128">If you follow this guidance but still reach the limits of the database, you may need to scale up the database.</span></span> <span data-ttu-id="c2897-129">此外，请考虑横向扩展，并将负载分区到不同的数据库服务器。</span><span class="sxs-lookup"><span data-stu-id="c2897-129">Also consider scaling horizontally and partitioning the load across database servers.</span></span> <span data-ttu-id="c2897-130">但是，分区可能需要重新设计应用程序。</span><span class="sxs-lookup"><span data-stu-id="c2897-130">However, partitioning may require redesigning the application.</span></span> <span data-ttu-id="c2897-131">有关详细信息，请参阅[数据分区][DataPartitioningGuidance]。</span><span class="sxs-lookup"><span data-stu-id="c2897-131">For more information, see [Data partitioning][DataPartitioningGuidance].</span></span>
