<span data-ttu-id="a1d22-104">本章介绍在微服务体系结构中管理数据时的注意事项。</span><span class="sxs-lookup"><span data-stu-id="a1d22-104">This chapter describes considerations for managing data in a microservices architecture.</span></span> <span data-ttu-id="a1d22-105">由于每个微服务管理自身的数据，因此，数据完整性和数据一致性是关键的挑战。</span><span class="sxs-lookup"><span data-stu-id="a1d22-105">Because every microservice manages its own data, data integrity and data consistency are critical challenges.</span></span>

<span data-ttu-id="a1d22-107">微服务的基本原则是每个服务管理其自身的数据。</span><span class="sxs-lookup"><span data-stu-id="a1d22-107">A basic principle of microservices is that each service manages its own data.</span></span> <span data-ttu-id="a1d22-108">两个服务不应共享数据存储。</span><span class="sxs-lookup"><span data-stu-id="a1d22-108">Two services should not share a data store.</span></span> <span data-ttu-id="a1d22-109">相反，每个服务负责管理维护自身的专用数据存储，其他服务不能直接访问该存储。</span><span class="sxs-lookup"><span data-stu-id="a1d22-109">Instead, each service is responsible for its own private data store, which other services cannot access directly.</span></span>

<span data-ttu-id="a1d22-110">制定此规则的原因是避免服务之间出现意外耦合 - 如果服务共享相同的基础数据架构，就会出现这种情况。</span><span class="sxs-lookup"><span data-stu-id="a1d22-110">The reason for this rule is to avoid unintentional coupling between services, which can result if services share the same underlying data schemas.</span></span> <span data-ttu-id="a1d22-111">如果对数据架构做了某项更改，必须在依赖于该数据库的每个服务之间协调该项更改。</span><span class="sxs-lookup"><span data-stu-id="a1d22-111">If there is a change to the data schema, the change must be coordinated across every service that relies on that database.</span></span> <span data-ttu-id="a1d22-112">通过隔离每个服务的数据存储，我们可以限制更改范围，并保持真正独立部署的灵活性。</span><span class="sxs-lookup"><span data-stu-id="a1d22-112">By isolating each service's data store, we can limit the scope of change, and preserve the agility of truly independent deployments.</span></span> <span data-ttu-id="a1d22-113">另一个原因是，每个微服务可能有自身的数据模型、查询或读/写模式。</span><span class="sxs-lookup"><span data-stu-id="a1d22-113">Another reason is that each microservice may have its own data models, queries, or read/write patterns.</span></span> <span data-ttu-id="a1d22-114">使用共享的数据存储会限制每个团队为其特定服务优化数据存储的能力。</span><span class="sxs-lookup"><span data-stu-id="a1d22-114">Using a shared data store limits each team's ability to optimize data storage for their particular service.</span></span>

<span data-ttu-id="a1d22-116">这种做法自然而然会导致 [Polyglot 持久性](https://martinfowler.com/bliki/PolyglotPersistence.html) &mdash; 在单个应用程序中使用多种数据存储技术。</span><span class="sxs-lookup"><span data-stu-id="a1d22-116">This approach naturally leads to [polyglot persistence](https://martinfowler.com/bliki/PolyglotPersistence.html) &mdash; the use of multiple data storage technologies within a single application.</span></span> <span data-ttu-id="a1d22-117">一个服务可能需要文档数据库的“读时架构”功能。</span><span class="sxs-lookup"><span data-stu-id="a1d22-117">One service might require the schema-on-read capabilities of a document database.</span></span> <span data-ttu-id="a1d22-118">另一个服务可能需要 RDBMS 提供的引用完整性。</span><span class="sxs-lookup"><span data-stu-id="a1d22-118">Another might need the referential integrity provided by an RDBMS.</span></span> <span data-ttu-id="a1d22-119">每个团队可以根据其服务任意做出最佳选择。</span><span class="sxs-lookup"><span data-stu-id="a1d22-119">Each team is free to make the best choice for their service.</span></span> <span data-ttu-id="a1d22-120">有关 Polyglot 持久性一般原则的详细信息，请参阅[使用最佳的数据存储完成作业](../guide/design-principles/use-the-best-data-store.md)。</span><span class="sxs-lookup"><span data-stu-id="a1d22-120">For more about the general principle of polyglot persistence, see [Use the best data store for the job](../guide/design-principles/use-the-best-data-store.md).</span></span>

## <a name="challenges"></a><span data-ttu-id="a1d22-123">挑战</span><span class="sxs-lookup"><span data-stu-id="a1d22-123">Challenges</span></span>

<span data-ttu-id="a1d22-124">这种分布式数据管理方法存在一些挑战。</span><span class="sxs-lookup"><span data-stu-id="a1d22-124">Some challenges arise from this distributed approach to managing data.</span></span> <span data-ttu-id="a1d22-125">首先，数据存储之间可能出现冗余，即同一个数据项显示在多个位置。</span><span class="sxs-lookup"><span data-stu-id="a1d22-125">First, there may be redundancy across the data stores, with the same item of data appearing in multiple places.</span></span> <span data-ttu-id="a1d22-126">例如，数据一开始存储为事务的一部分，然后又存储在其他位置供分析、报告或存档。</span><span class="sxs-lookup"><span data-stu-id="a1d22-126">For example, data might be stored as part of a transaction, then stored elsewhere for analytics, reporting, or archiving.</span></span> <span data-ttu-id="a1d22-127">重复或分区的数据可能导致数据完整性和一致性问题。</span><span class="sxs-lookup"><span data-stu-id="a1d22-127">Duplicated or partitioned data can lead to issues of data integrity and consistency.</span></span> <span data-ttu-id="a1d22-128">当数据关系跨越多个服务时，无法使用传统的数据管理方法来实施关系。</span><span class="sxs-lookup"><span data-stu-id="a1d22-128">When data relationships span multiple services, you can't use traditional data management techniques to enforce the relationships.</span></span>

<span data-ttu-id="a1d22-129">传统数据建模使用“一个位置一个事实”的规则。</span><span class="sxs-lookup"><span data-stu-id="a1d22-129">Traditional data modeling uses the rule of "one fact in one place."</span></span> <span data-ttu-id="a1d22-130">每个实体只在架构中出现一次。</span><span class="sxs-lookup"><span data-stu-id="a1d22-130">Every entity appears exactly once in the schema.</span></span> <span data-ttu-id="a1d22-131">其他实体可以保留对该实体的引用，但不能复制它。</span><span class="sxs-lookup"><span data-stu-id="a1d22-131">Other entities may hold references to it but not duplicate it.</span></span> <span data-ttu-id="a1d22-132">传统方法的明显优势是在一个位置进行更新，从而可以避免数据一致性问题。</span><span class="sxs-lookup"><span data-stu-id="a1d22-132">The obvious advantage to the traditional approach is that updates are made in a single place, which avoids problems with data consistency.</span></span> <span data-ttu-id="a1d22-133">在微服务体系结构中，必须考虑如何在服务之间传播更新，以及在未实施强一致性的情况下，如果数据出现在多个位置，应如何管理最终一致性。</span><span class="sxs-lookup"><span data-stu-id="a1d22-133">In a microservices architecture, you have to consider how updates are propagated across services, and how to manage eventual consistency when data appears in multiple places without strong consistency.</span></span>

## <a name="approaches-to-managing-data"></a><span data-ttu-id="a1d22-134">数据管理方法</span><span class="sxs-lookup"><span data-stu-id="a1d22-134">Approaches to managing data</span></span>

<span data-ttu-id="a1d22-135">不存在以一应百的方法，不过，在微服务体系结构中管理数据时，可以遵循一些通用准则。</span><span class="sxs-lookup"><span data-stu-id="a1d22-135">There is no single approach that's correct in all cases, but here are some general guidelines for managing data in a microservices architecture.</span></span>

- <span data-ttu-id="a1d22-136">如果可能，请采用最终一致性。</span><span class="sxs-lookup"><span data-stu-id="a1d22-136">Embrace eventual consistency where possible.</span></span> <span data-ttu-id="a1d22-137">了解系统中需要强一致性或 ACID 事务的位置，以及可接受最终一致性的位置。</span><span class="sxs-lookup"><span data-stu-id="a1d22-137">Understand the places in the system where you need strong consistency or ACID transactions, and the places where eventual consistency is acceptable.</span></span>

- <span data-ttu-id="a1d22-138">需要强一致性保证时，一个服务可以代表给定实体的事实源（通过 API 公开）。</span><span class="sxs-lookup"><span data-stu-id="a1d22-138">When you need strong consistency guarantees, one service may represent the source of truth for a given entity, which is exposed through an API.</span></span> <span data-ttu-id="a1d22-139">其他服务可以保留自身的数据副本或数据子集，这些数据与主数据最终保持一致，但不被视为事实源。</span><span class="sxs-lookup"><span data-stu-id="a1d22-139">Other services might hold their own copy of the data, or a subset of the data, that is eventually consistent with the master data but not considered the source of truth.</span></span> <span data-ttu-id="a1d22-140">例如，假设有一个包含客户订单服务和推荐服务的电子商务系统。</span><span class="sxs-lookup"><span data-stu-id="a1d22-140">For example, imagine an e-commerce system with a customer order service and a recommendation service.</span></span> <span data-ttu-id="a1d22-141">推荐服务可以侦听来自订单服务的事件，但如果客户请求退款，则只有订单服务包含完整的交易历史记录，而推荐服务则没有此类信息。</span><span class="sxs-lookup"><span data-stu-id="a1d22-141">The recommendation service might listen to events from the order service, but if a customer requests a refund, it is the order service, not the recommendation service, that has the complete transaction history.</span></span>

- <span data-ttu-id="a1d22-142">对于事务，可以使用[计划程序代理监督程序](../patterns/scheduler-agent-supervisor.md)和[补偿事务](../patterns/compensating-transaction.md)等模式来保持多个服务之间的数据一致性。</span><span class="sxs-lookup"><span data-stu-id="a1d22-142">For transactions, use patterns such as [Scheduler Agent Supervisor](../patterns/scheduler-agent-supervisor.md) and [Compensating Transaction](../patterns/compensating-transaction.md) to keep data consistent across several services.</span></span>  <span data-ttu-id="a1d22-143">可能需要存储附加的数据片段用于捕获跨多个服务的工作单元的状态，以免多个服务之间发生部分失败。</span><span class="sxs-lookup"><span data-stu-id="a1d22-143">You may need to store an additional piece of data that captures the state of a unit of work that spans multiple services, to avoid partial failure among multiple services.</span></span> <span data-ttu-id="a1d22-144">例如，在处理多步骤事务的过程中，可将某个工作项保留在持久队列中。</span><span class="sxs-lookup"><span data-stu-id="a1d22-144">For example, keep a work item on a durable queue while a multi-step transaction is in progress.</span></span>

- <span data-ttu-id="a1d22-145">只存储服务所需的数据。</span><span class="sxs-lookup"><span data-stu-id="a1d22-145">Store only the data that a service needs.</span></span> <span data-ttu-id="a1d22-146">服务可能只需要有关域实体的一部分信息。</span><span class="sxs-lookup"><span data-stu-id="a1d22-146">A service might only need a subset of information about a domain entity.</span></span> <span data-ttu-id="a1d22-147">例如，在“交货”边界上下文中，我们需要知道哪个客户已关联到特定的交付项。</span><span class="sxs-lookup"><span data-stu-id="a1d22-147">For example, in the Shipping bounded context, we need to know which customer is associated to a particular delivery.</span></span> <span data-ttu-id="a1d22-148">但是，我们不需要客户的帐单地址 &mdash; 该信息由“帐户”边界上下文管理。</span><span class="sxs-lookup"><span data-stu-id="a1d22-148">But we don't need the customer's billing address &mdash; that's managed by the Accounts bounded context.</span></span> <span data-ttu-id="a1d22-149">在此情况下，认真考虑域的要求并使用 DDD 方法可能有所帮助。</span><span class="sxs-lookup"><span data-stu-id="a1d22-149">Thinking carefully about the domain, and using a DDD approach, can help here.</span></span>

- <span data-ttu-id="a1d22-150">考虑服务是否是内聚且松散耦合的。</span><span class="sxs-lookup"><span data-stu-id="a1d22-150">Consider whether your services are coherent and loosely coupled.</span></span> <span data-ttu-id="a1d22-151">如果两个服务持续相互交换信息，导致出现琐碎 API，则你可能需要通过合并两个服务或重构其功能来重绘服务边界。</span><span class="sxs-lookup"><span data-stu-id="a1d22-151">If two services are continually exchanging information with each other, resulting in chatty APIs, you may need to redraw your service boundaries, by merging two services or refactoring their functionality.</span></span>

- <span data-ttu-id="a1d22-152">使用[事件驱动的体系结构样式](../guide/architecture-styles/event-driven.md)。</span><span class="sxs-lookup"><span data-stu-id="a1d22-152">Use an [event driven architecture style](../guide/architecture-styles/event-driven.md).</span></span> <span data-ttu-id="a1d22-153">在此体系结构样式中，当某个服务的公共模型或实体发生更改时，该服务会发布事件。</span><span class="sxs-lookup"><span data-stu-id="a1d22-153">In this architecture style, a service publishes an event when there are changes to its public models or entities.</span></span> <span data-ttu-id="a1d22-154">相关的服务可以订阅这些事件。</span><span class="sxs-lookup"><span data-stu-id="a1d22-154">Interested services can subscribe to these events.</span></span> <span data-ttu-id="a1d22-155">例如，另一个服务可以使用事件来构造更适合用于查询的具体化数据视图。</span><span class="sxs-lookup"><span data-stu-id="a1d22-155">For example, another service could use the events to construct a materialized view of the data that is more suitable for querying.</span></span>

- <span data-ttu-id="a1d22-156">拥有事件的服务应发布可用于自动序列化和反序列化事件的架构，以免发布方与订阅方之间发生紧密耦合。</span><span class="sxs-lookup"><span data-stu-id="a1d22-156">A service that owns events should publish a schema that can be used to automate serializing and deserializing the events, to avoid tight coupling between publishers and subscribers.</span></span> <span data-ttu-id="a1d22-157">考虑使用 JSON 架构，或者 [Microsoft Bond](https://github.com/Microsoft/bond)、Protobuf 或 Avro 等框架。</span><span class="sxs-lookup"><span data-stu-id="a1d22-157">Consider JSON schema or a framework like [Microsoft Bond](https://github.com/Microsoft/bond), Protobuf, or Avro.</span></span>

- <span data-ttu-id="a1d22-158">在大规模运营中，事件可能成为系统中的瓶颈，因此，请考虑使用聚合或批处理来减少总负载。</span><span class="sxs-lookup"><span data-stu-id="a1d22-158">At high scale, events can become a bottleneck on the system, so consider using aggregation or batching to reduce the total load.</span></span>

## <a name="drone-delivery-choosing-the-data-stores"></a><span data-ttu-id="a1d22-159">无人机交付：选择数据存储</span><span class="sxs-lookup"><span data-stu-id="a1d22-159">Drone Delivery: Choosing the data stores</span></span>

<span data-ttu-id="a1d22-160">尽管只包含少量的几个服务，但“交货”边界上下文仍然演示了本部分中所述的要点。</span><span class="sxs-lookup"><span data-stu-id="a1d22-160">Even with only a few services, the Shipping bounded context illustrates several of the points discussed in this section.</span></span>

<span data-ttu-id="a1d22-161">当用户安排新的交付时，客户端请求中将会包含有关交付的信息（例如收件和投递地点）和有关包裹的信息（例如大小与重量）。</span><span class="sxs-lookup"><span data-stu-id="a1d22-161">When a user schedules a new delivery, the client request includes information about the both the delivery, such as the pickup and dropoff locations, and about the package, such as the size and weight.</span></span> <span data-ttu-id="a1d22-162">此信息定义工作单位，引入服务会将其发送到事件中心。</span><span class="sxs-lookup"><span data-stu-id="a1d22-162">This information defines a unit of work, which the Ingestion service sends to Event Hubs.</span></span> <span data-ttu-id="a1d22-163">当计划程序服务执行工作流时，该工作单位必须保留在持久性存储中，以便不会丢失交付请求。</span><span class="sxs-lookup"><span data-stu-id="a1d22-163">It's important that the unit of work stays in persistent storage while the Scheduler service is executing the workflow, so that no delivery requests are lost.</span></span> <span data-ttu-id="a1d22-164">有关工作流的更多讨论，请参阅[引入和工作流](./ingestion-workflow.md)。</span><span class="sxs-lookup"><span data-stu-id="a1d22-164">For more discussion of the workflow, see [Ingestion and workflow](./ingestion-workflow.md).</span></span>

### <a name="delivery-service"></a><span data-ttu-id="a1d22-166">交付服务</span><span class="sxs-lookup"><span data-stu-id="a1d22-166">Delivery service</span></span>

<span data-ttu-id="a1d22-167">交付服务存储有关当前已安排的或正在处理的每项交付的信息。</span><span class="sxs-lookup"><span data-stu-id="a1d22-167">The Delivery service stores information about every delivery that is currently scheduled or in progress.</span></span> <span data-ttu-id="a1d22-168">它会侦听来自无人机的事件，并跟踪正在进行的交付的状态。</span><span class="sxs-lookup"><span data-stu-id="a1d22-168">It listens for events from the drones, and tracks the status of deliveries that are in progress.</span></span> <span data-ttu-id="a1d22-169">此外，它会发送包含交付状态更新内容的域事件。</span><span class="sxs-lookup"><span data-stu-id="a1d22-169">It also sends domain events with delivery status updates.</span></span>

<span data-ttu-id="a1d22-170">用户在等待包裹期间，预期会经常检查交付状态。</span><span class="sxs-lookup"><span data-stu-id="a1d22-170">It's expected that users will frequently check the status of a delivery while they are waiting for their package.</span></span> <span data-ttu-id="a1d22-171">因此，交付服务需要一个具有突出吞吐量（读取和写入）而不是持久性的数据存储。</span><span class="sxs-lookup"><span data-stu-id="a1d22-171">Therefore, the Delivery service requires a data store that emphasizes throughput (read and write) over long-term storage.</span></span> <span data-ttu-id="a1d22-172">此外，交付服务不会执行任何复杂的查询或分析，而只是提取给定交付的最新状态。</span><span class="sxs-lookup"><span data-stu-id="a1d22-172">Also, the Delivery service does not perform any complex queries or analysis, it simply fetches the latest status for a given delivery.</span></span> <span data-ttu-id="a1d22-173">交付服务团队可以选择 Azure Redis 缓存来提高读写性能。</span><span class="sxs-lookup"><span data-stu-id="a1d22-173">The Delivery service team chose Azure Redis Cache for its high read-write performance.</span></span> <span data-ttu-id="a1d22-174">Redis 中存储的信息的生存期相对较短。</span><span class="sxs-lookup"><span data-stu-id="a1d22-174">The information stored in Redis is relatively short-lived.</span></span> <span data-ttu-id="a1d22-175">交付完成后，交付历史记录服务即是记录系统。</span><span class="sxs-lookup"><span data-stu-id="a1d22-175">Once a delivery is complete, the Delivery History service is the system of record.</span></span>

### <a name="delivery-history-service"></a><span data-ttu-id="a1d22-176">交付历史记录服务</span><span class="sxs-lookup"><span data-stu-id="a1d22-176">Delivery History service</span></span>
